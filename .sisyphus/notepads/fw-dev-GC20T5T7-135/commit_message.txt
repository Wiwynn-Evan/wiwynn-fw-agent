grandcanyon2: enable BIC health monitoring with fault tolerance

[Issue Description]
- Related to GC20T5T7-135
- BIC health monitoring is disabled on GC2.0 platform (bic_health.enabled = false
  in healthd-config.json). When enabled without PAL layer fixes, healthd may
  falsely trigger BIC hardware reset during early boot (KV store not yet
  initialized) or after a legitimate BIC reset (BIC still rebooting).

[Root Cause]
- pal_is_bic_heartbeat_ok() returns false when kv_get() fails (KV key not yet
  created by front-paneld), causing healthd to misinterpret as heartbeat failure.
  Three consecutive failures (3 x 60s = 180s) trigger pal_bic_hw_reset().
- After BIC hardware reset, there is no cooldown period. BIC takes 30-60 seconds
  to boot, during which heartbeat KV is not updated. Once is_already_reset is
  cleared upon brief recovery, the reset cycle can repeat indefinitely.

[Solution]
- Enable bic_health in GC2.0 healthd-config.json (enabled: false -> true).
- Fix KV store fault tolerance: pal_is_bic_heartbeat_ok() now returns true
  (tolerant) when kv_get() fails, with LOG_WARNING for debug visibility.
  This is safe because pal_is_bic_ready() and pal_bic_self_test() are checked
  before heartbeat in healthd's bic_health_monitor thread.
- Add 120-second post-reset cooldown: pal_bic_hw_reset() records reset timestamp,
  pal_is_bic_heartbeat_ok() skips heartbeat check during cooldown window to allow
  BIC to fully boot before resuming monitoring.

   $ log-util all --print | grep "BIC HW RESET"
   (After reset: verify no second reset within 120 seconds)
